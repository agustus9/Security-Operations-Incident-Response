# 🧪 Adversarial Simulation & Detection Validation

## 📋 Scenario & Business Problem
A Security Operations Center cannot assume its detection rules work until they are actively tested against real-world attack vectors. If an endpoint logging agent fails or a SIEM ingestion pipeline breaks, a critical security event will go unnoticed. 

The objective of this lab is to conduct controlled, safe attack simulations (such as automated password guessing and web app testing) in an isolated sandbox environment. By mimicking adversary behavior, we map out the specific logs generated by the attacks to confirm that our security systems provide adequate visibility and alert accuracy.

## 💡 Solution Architecture
This project documents a three-phase detection validation workflow:
1. **Adversarial Execution:** Running standard attack commands against test targets using accessible network diagnostics tools (Hydra, Metasploit, BeEF).
2. **Telemetry Verification:** Checking the raw system event logs to ensure the malicious behavior was recorded with correct attributes.
3. **Alert Validation:** Matching the generated logs against our custom SIEM queries to verify that security dashboards trigger correctly, reducing baseline alert fatigue.

## 🏗️ Technical Components
* **Simulation Tooling:** Hydra Brute-Force Engine, Metasploit Framework, BeEF Browser Integration
* **Validation Environment:** Splunk Enterprise, Microsoft Sentinel, Isolated Virtual Network Labs
* **Framework Mapping:** MITRE ATT&CK (Credential Access: T1110 - Brute Force, Discovery: T1046 - Network Service Discovery)

## 🛠️ Simulation Scenarios & Log Baselines

### Scenario 1: Automated SSH Brute Force via Hydra
* **Attack Command executed:** `hydra -l testuser -P passwords.txt ssh://192.168.1.50 -t 4`
* **Log Evidence Generated:** Linux `secure` / `auth.log` system trails.
* **Telemetry Validation String:** Look for high-frequency repetitions of `Failed password for testuser from [IP] port [Port] ssh2`.
* **SIEM Verification Rule (Splunk SPL):**
```splunk
sourcetype=linux_secure "Failed password"

| stats count as AttemptCount by src_ip, user
| where AttemptCount > 10
```

### Scenario 2: Vulnerability Scanning via Metasploit
* **Attack Module executed:** `use auxiliary/scanner/portscan/tcp`
* **Log Evidence Generated:** Network firewall connection logs (`sourcetype=fortigate_traffic`).
* **Telemetry Validation String:** A single source IP address initiating thousands of rapid connection requests across consecutive destination ports in a short window.
* **SIEM Verification Rule (Sentinel KQL):**
```kusto
SigninLogs // or NetworkTrafficLogs depending on your dataset

| where ResultType == "50126" or Action == "blocked"
| summarize TargetPortsScanned = dcount(DestinationPort) by SourceIP
| where TargetPortsScanned > 20
```

### Scenario 3: Browser Framework Injections via BeEF
* **Attack Vector:** Injecting a JavaScript hook string into an unauthenticated internal web application field.
* **Log Evidence Generated:** Web server access logs (`sourcetype=apache_access` or `IIS`).
* **Telemetry Validation String:** HTTP requests containing raw execution terms, anomalous scripts, or uncommon client browser agent details.
* **SIEM Verification Rule (QRadar AQL):**
```aql
SELECT username, sourceip, payload FROM events 
WHERE (payload CONTAINS 'script' OR payload CONTAINS 'hook.js')
```

## 🚀 How to Utilize This Lab for Detection Engineering
1. **Freeze Baseline:** Ensure normal, authorized user baseline metrics are established before running simulations.
2. **Execute & Timestamp:** Run the chosen attack simulation script and record the exact system execution timestamp.
3. **Log Hunt:** Search your SIEM environment for the exact timestamp window to verify that your data connectors properly captured the event.
4. **Tune Alerts:** Adjust your detection thresholds (e.g., changing failure limits from 5 to 10) to eliminate background false positives.
