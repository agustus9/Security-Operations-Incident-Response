# 🧪 Adversarial Simulation & Detection Validation

## 📋 Scenario & Business Problem
A Security Operations Center cannot assume its detection rules work until they are actively tested against live adversarial tactics. If an endpoint logging agent fails, a cloud connector disconnects, or a SIEM ingestion pipeline breaks, a critical breach will go completely unnoticed. 

The objective of this lab is to conduct controlled, safe attack simulations inside an isolated sandbox network. By mimicking real-world attacker techniques, we can map out the exact logs generated by the exploits and validate that our alerting thresholds trigger accurately across multiple SIEM engines, directly reducing corporate alert fatigue.

## 💡 Solution Architecture
This project establishes a standardized, repeatable validation framework:
1. **Adversarial Execution:** Running standard attack commands against test targets using accessible security tools (Hydra, Metasploit, BeEF).
2. **Multi-SIEM Log Verification:** Checking the raw system event logs across **Splunk**, **Sentinel**, and **QRadar** to verify the malicious behavior was recorded with the correct metadata.
3. **Alert Tuning:** Optimizing query thresholds to ensure high-fidelity alerting while filtering out background false positives.

## 🏗️ Technical Components
* **Simulation Tooling:** Hydra Brute-Force Engine, Metasploit Framework, BeEF Browser Integration
* **Ingestion Validation:** Splunk Enterprise, Microsoft Sentinel Workspace, IBM QRadar Event Console
* **Framework Mapping:** MITRE ATT&CK (Credential Access: T1110 - Brute Force, Discovery: T1046 - Network Service Discovery)

## 🛠️ Simulation Scenarios & Multi-SIEM Validation

### Lab Scenario 1: Automated Password Spraying via Hydra
* **Adversarial Action:** Executing a high-velocity automated password guessing attack against an internal SSH asset.
* **Execution Command:** `hydra -l targetadmin -P rockyou.txt ssh://192.168.1.50 -t 4`
* **Target Telemetry Generated:** Linux system authentication trails (`/var/log/auth.log`).

#### ➡️ Splunk Enterprise Validation (SPL)
```splunk
sourcetype=linux_secure "Failed password" 

| stats count as AttemptCount by src_ip, user 
| where AttemptCount > 10 
| sort - AttemptCount
```

#### ➡️ Microsoft Sentinel Validation (KQL)
```kusto
Syslog

| where ProcessName == "sshd" and Message contains "Failed password"
| summarize AttemptCount = count() by Computer, HostIP, ProcessName
| where AttemptCount > 10
```

#### ➡️ IBM QRadar Validation (AQL)
```aql
SELECT sourceip AS 'Attacker IP', username AS 'Target User', count(*) AS 'Failure Count' 
FROM events 
WHERE qid = 5000012 OR payload CONTAINS 'Failed password' 
GROUP BY sourceip, username 
HAVING "Failure Count" > 10
```

### Lab Scenario 2: Network Service & Port Scanning via Metasploit
* **Adversarial Action:** Mapping an internal network segment to identify live services and unpatched application ports.
* **Execution Module:** `use auxiliary/scanner/portscan/tcp` (Targeting network ranges)
* **Target Telemetry Generated:** Enterprise firewall connection logs.

#### ➡️ Splunk Enterprise Validation (SPL)
```splunk
index=network sourcetype=firewall_traffic action=blocked 

| stats dcount(dest_port) as ScannedPorts by src_ip 
| where ScannedPorts > 20
```

#### ➡️ Microsoft Sentinel Validation (KQL)
```kusto
CommonSecurityLog

| where DeviceAction == "deny" or DeviceAction == "blocked"
| summarize ScannedPorts = dcount(DestinationPort) by SourceIP
| where ScannedPorts > 20
```

#### ➡️ IBM QRadar Validation (AQL)
```aql
SELECT sourceip AS 'Scanner IP', COUNT(DISTINCT destinationport) AS 'Scanned Ports' 
FROM events 
WHERE (payload CONTAINS 'deny' OR payload CONTAINS 'blocked') 
GROUP BY sourceip 
HAVING "Scanned Ports" > 20
```

### Lab Scenario 3: Hook Injection & Cross-Site Scripting via BeEF
* **Adversarial Action:** Injecting a malicious JavaScript hook string into an unauthenticated internal web application field to control client browsers.
* **Execution Vector:** Exploiting a web parameter field with `http://192.168.1`.
* **Target Telemetry Generated:** Web application access and execution server logs.

#### ➡️ Splunk Enterprise Validation (SPL)
```splunk
index=web sourcetype=apache_access OR sourcetype=iis_logs "hook.js" OR "<script>"

| table _time, src_ip, uri_path, status, user_agent
```

#### ➡️ Microsoft Sentinel Validation (KQL)
```kusto
W3CIISLog
| where csUriQuery contains "hook.js" or csUriQuery contains "script"
| project TimeGenerated, cIP, csUriStem, csUriQuery, scStatus
```

#### ➡️ IBM QRadar Validation (AQL)
```aql
SELECT sourceip AS 'Client IP', DATEFORMAT(starttime, 'yyyy-MM-dd HH:mm:ss') AS 'Time', payload AS 'Raw Request' 
FROM events 
WHERE (payload CONTAINS 'hook.js' OR payload CONTAINS '<script>')
```

## 🚀 Detection Engineering & Tuning Framework
* **Step 1 (Baseline Verification):** Check normal, authorized network traffic metrics before executing tests to ensure zero operational interference.
* **Step 2 (Simulate & Timestamp):** Run the chosen attack vector and record the exact system execution timestamp down to the second.
* **Step 3 (SIEM Audit Logs):** Query your respective Splunk, Sentinel, and QRadar platforms using the intermediate code blocks above to verify data ingestion paths.
* **Step 4 (Threshold Optimization):** Adjust failure counts and time windows to filter out normal network noise, ensuring alerts trigger only on true positive malicious actions.
